# 面试问题准备
## 你这份工作做了什么？负责哪些模块？
我在公司主要负责键盘功能检测软件这个项目。
公司是一人负责一个项目，方便后续算绩效、评奖金
先是整理项目立项相关内容，理清项目背景和要实现的目标，编写立项书；然后定好整个项目用什么技术、排好开发时间计划，编写产品计划书；之后进行软件开发、功能测试，测试完成后把软件部署到厂区投入使用；最后再收集用户的使用反馈，进行软件优化
## 用到什么技术栈？解决过什么实际问题？
我这个项目主要用的技术栈是 Qt C++ 做界面和业务开发，用 CMake 来构建项目，代码版本管理用的是公司内部的 GitLab。
实际工作中解决的问题是：之前产线一直用第三方的 KeyboardTest 键盘检测软件，软件兼容性很好，台式机和笔记本都能使用。产线生产过程中工人有自己的 SOP ，需要在软件里加入对应的防呆逻辑，用来降低外租设备的 CDOA ，但是第三方软件是闭源的，无法加入新的逻辑，不满足产线的使用要求。
所以项目主要内容就是开发键盘检测工具，最终目标就是降低外租设备的 CDOA 不良率。我在软件中做的防呆逻辑是：统计每个按键的有效按压次数，成功按压 4 次，才判定这个按键功能正常；同时配合优化工人 SOP，要求按压按键三个角点加中心点进行测试，从软件逻辑和操作流程两方面一起降低产品 CDOA 不良率。
## 为什么干了 4 个月就离职？
我当时把键盘检测软件项目做到后期，正处在收集用户反馈、做软件优化的阶段。
这时候公司管理层做了一次比较大的调整，把我们原来的研发测试组，重新划分成软件组和硬件组。我和几个一起入职的同事，被统一分到了硬件组。
当初校招招聘时，岗位要求是计算机相关专业，整体招录也是按软件研发测试方向进行的，我和几个同事都是计算机科学与技术专业毕业，都被分配到了研发测试岗的硬件组。
但是，硬件组要求必须懂数电、模电，还要会画 PCB、做硬件相关开发，这些都不是我的专业方向，也和我规划的职业道路不匹配。
如果继续留在公司，就得从零开始学电路和硬件基础，学习成本很高，也偏离了我的职业规划，因此最终选择了离职。
## 项目为什么选用 Qt C++ 开发？为什么不用其他 GUI 框架？
这个键盘检测工具主要运行在 Windows 笔记本和台式机上，对软件的运行稳定性、按键响应速度、设备兼容性要求很高，所以最终选用了 Qt C++ 来开发。
1. 键盘检测需要实时捕获按键状态、处理底层输入事件，Qt 结合 C++ 能直接跟系统硬件层做交互，事件响应延迟低、执行效率高，很适合做这种实时检测类工具。
2. 产线里有不同品牌、不同系统版本的笔记本和台式机，Qt 兼容性强，一套代码能稳定运行在各种 Windows 版本上，不用重复开发。
3. Qt 自带成熟的界面库、信号槽机制，开发可视化键盘界面、状态刷新非常高效，适合开发工具。
4. 而MFC框架老旧，UI开发繁琐，开发成本高；WinForm 和 WPF 虽然也是 Windows 桌面框架，但它们只支持 Windows 平台，在底层硬件交互、外设检测方面不如 Qt 灵活。
## 为什么用 CMake，不用 Qt 自带的 qmake？你是怎么配置 CMake 的？
项目用 CMake 而不是 qmake，主要是因为 CMake 跨平台支持更好、项目管理更灵活、生态更强大，而且现在新版 Qt 默认推荐 CMake。它对模块划分、第三方库集成、编译链适配都比 qmake 更强大，方便后续项目扩展和维护。
在 CMakeLists.txt 里指定了 CMake 最低版本、项目名称，设置 C++ 标准，编译模式用 Release 发布版本；打开 Qt 相关的自动编译开关，AUTOMOC、AUTORCC、AUTOUIC 都开启，让 Qt 的信号槽、UI 文件可以自动编译；指定 Qt 的安装路径，用 find_package 找到 Qt6 Core 和 Widgets 这两个核心模块；把项目的源文件、头文件统一管理，通过 target_include_directories 指定头文件路径；最后链接 Qt 库。
